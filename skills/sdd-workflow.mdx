# SDD Workflow Skills

The SDD (Spec-Driven Development) workflow is implemented through **9 specialized skills** that guide AI agents through each phase of structured feature development.

## Workflow Overview

SDD follows this sequence:

```
┌────────────┐
│  sdd-init   │  Initialize project context
└─────┬──────┘
     │
     │  When starting a new change:
     │
┌────┴────────┐
│ sdd-explore  │  (Optional) Explore problem space
└─────┬───────┘
     │
┌────┴────────┐
│ sdd-propose │  Define intent, scope, approach
└─────┬───────┘
     │
┌────┴────────┐
│  sdd-spec   │  Write behavioral specifications
└─────┬───────┘
     │
┌────┴────────┐
│ sdd-design  │  Make architectural decisions
└─────┬───────┘
     │
┌────┴────────┐
│ sdd-tasks   │  Break into implementable chunks
└─────┬───────┘
     │
┌────┴────────┐
│ sdd-apply   │  Write actual code
└─────┬───────┘
     │
┌────┴────────┐
│ sdd-verify  │  Confirm implementation matches specs
└─────┬───────┘
     │
┌────┴─────────┐
│ sdd-archive │  Merge and clean up
└─────────────┘
```

Each phase produces an **artifact** that feeds into the next phase.

## Phase Details

### sdd-init

**Purpose**: Initialize SDD context in any project

**Trigger**: When user wants to start using SDD, or says "sdd init", "iniciar sdd", "openspec init"

**What it does**:
1. Detects project tech stack (package.json, go.mod, pyproject.toml, etc.)
2. Identifies existing conventions (linters, test frameworks, CI)
3. Bootstraps persistence backend based on mode
4. Generates skill registry (scans user + project skills)
5. Persists project context

**Artifacts created**:

**Engram mode**:
```typescript
mem_save(
  title: "sdd-init/{project-name}",
  topic_key: "sdd-init/{project-name}",
  type: "architecture",
  project: "{project-name}",
  content: "{detected project context}"
)
```

**OpenSpec mode**:
```
openspec/
├── config.yaml              # Project-specific SDD config
├── specs/                   # Ready for specifications
└── changes/                 # Ready for change proposals
    └── archive/             # Completed changes
```

**Both modes**:
```
.atl/skill-registry.md       # Catalog of available skills
```

**Key rules**:
- NEVER create placeholder spec files
- ALWAYS detect real tech stack (don't guess)
- If `openspec/` already exists, report and ask before updating
- Keep config.yaml context CONCISE (max 10 lines)

### sdd-explore

**Purpose**: Explore problem space and recommend approach

**Trigger**: When user wants to understand a topic before making changes

**What it does**:
1. Reads existing code in relevant areas
2. Identifies patterns and conventions
3. Finds similar existing implementations
4. Analyzes dependencies and constraints
5. Recommends approach with trade-offs

**Artifacts created**:

**Engram mode**:
```typescript
mem_save(
  title: "sdd/{change-name}/exploration",
  topic_key: "sdd/{change-name}/exploration",
  type: "architecture",
  project: "{project}",
  content: "# Exploration: {topic}\n\n..."
)
```

**OpenSpec mode**:
```
openspec/changes/{change-name}/exploration.md
```

**Key rules**:
- Read code BEFORE making recommendations
- Compare multiple approaches with trade-offs
- Identify existing patterns to follow
- Note any breaking changes or risks

### sdd-propose

**Purpose**: Create a change proposal with intent, scope, and approach

**Trigger**: When orchestrator launches you to create/update a proposal

**What it does**:
1. Loads exploration analysis (if exists)
2. Reads existing specs in affected areas
3. Creates `proposal.md` with structured sections
4. Identifies affected code areas
5. Defines success criteria and rollback plan

**Artifacts created**:

**Engram mode**:
```typescript
mem_save(
  title: "sdd/{change-name}/proposal",
  topic_key: "sdd/{change-name}/proposal",
  type: "architecture",
  project: "{project}",
  content: "# Proposal: {title}\n\n..."
)
```

**OpenSpec mode**:
```
openspec/changes/{change-name}/proposal.md
```

**Proposal structure**:
```markdown
# Proposal: Add Dark Mode

## Intent
What problem are we solving? Why does this change need to happen?

## Scope
### In Scope
- Concrete deliverable 1
- Concrete deliverable 2

### Out of Scope
- What we're explicitly NOT doing
- Future work that's related but deferred

## Approach
High-level technical approach. Reference exploration recommendations.

## Affected Areas
| Area | Impact | Description |
|------|--------|-------------|
| path/to/area | New/Modified/Removed | What changes |

## Risks
| Risk | Likelihood | Mitigation |
|------|------------|------------|

## Rollback Plan
How to revert if something goes wrong. Be specific.

## Success Criteria
- [ ] How do we know this change succeeded?
```

**Key rules**:
- Every proposal MUST have a rollback plan
- Every proposal MUST have success criteria
- Use concrete file paths in "Affected Areas"
- Keep proposal CONCISE (it's a thinking tool, not a novel)

### sdd-spec

**Purpose**: Write behavioral specifications using Given/When/Then scenarios

**Trigger**: When orchestrator launches you to write specs for a change

**What it does**:
1. Reads proposal to understand intent
2. Breaks functionality into testable scenarios
3. Writes specs using Given/When/Then format
4. Uses RFC 2119 keywords (MUST, SHALL, SHOULD, MAY)
5. Defines acceptance criteria

**Artifacts created**:

**Engram mode**:
```typescript
mem_save(
  title: "sdd/{change-name}/spec",
  topic_key: "sdd/{change-name}/spec",
  type: "architecture",
  project: "{project}",
  content: "# Specification: {domain}\n\n..."
)
```

**OpenSpec mode**:
```
openspec/changes/{change-name}/specs/{domain}/spec.md
```

**Spec structure**:
```markdown
# Specification: Dark Mode Toggle

## Scenarios

### Scenario: User enables dark mode

**Given** the user is on the settings page
**And** dark mode is currently disabled
**When** the user clicks the theme toggle
**Then** the UI MUST switch to dark theme
**And** the preference MUST be saved to localStorage
**And** the toggle MUST show "enabled" state

### Scenario: Dark mode persists across sessions

**Given** the user has enabled dark mode
**When** the user closes and reopens the app
**Then** the app MUST load in dark theme
**And** the toggle MUST show "enabled" state

## Edge Cases

### Scenario: localStorage is unavailable

**Given** localStorage is blocked by browser settings
**When** the user toggles dark mode
**Then** the theme SHOULD change in the current session
**But** the app MAY fall back to system default on reload
```

**Key rules**:
- Use Given/When/Then for EVERY scenario
- Use RFC 2119 keywords: MUST (required), SHALL (required), SHOULD (recommended), MAY (optional)
- Test one behavior per scenario
- Include edge cases and error states

### sdd-design

**Purpose**: Make architectural decisions with rationale

**Trigger**: When orchestrator launches you to design the solution

**What it does**:
1. Reads proposal and specs
2. Makes technical decisions (data structures, APIs, patterns)
3. Documents rationale for each decision
4. Creates sequence diagrams for complex flows
5. Identifies required changes to existing code

**Artifacts created**:

**Engram mode**:
```typescript
mem_save(
  title: "sdd/{change-name}/design",
  topic_key: "sdd/{change-name}/design",
  type: "architecture",
  project: "{project}",
  content: "# Design: {change-name}\n\n..."
)
```

**OpenSpec mode**:
```
openspec/changes/{change-name}/design.md
```

**Design structure**:
```markdown
# Design: Dark Mode Toggle

## Architecture Decisions

### Decision 1: CSS Variables for Theming

**Decision**: Use CSS custom properties on `:root` element

**Rationale**:
- Zero runtime overhead (pure CSS)
- Works with existing styles (no refactor required)
- Standard approach (MDN recommends)

**Alternatives considered**:
- Tailwind dark mode classes: Requires rewriting all styles
- JS-based theme switching: Adds runtime cost, flash of wrong theme

### Decision 2: localStorage for Persistence

**Decision**: Store theme preference in localStorage as "theme" key

**Rationale**:
- Simple, widely supported
- No server dependency
- Fast read on page load

**Alternatives considered**:
- Cookies: Unnecessary for client-only preference
- Server-side storage: Overkill, requires auth

## Data Structures

```typescript
type Theme = 'light' | 'dark';

interface ThemeConfig {
  current: Theme;
  default: Theme;
  systemPreference: Theme | null;
}
```

## Sequence Diagram

```
User         ToggleButton      ThemeService      LocalStorage
  |──click──────>|                   |                 |
  |                |----toggle()----->|                 |
  |                |                  |---getItem(theme)---->|
  |                |                  |<--current='light'----|
  |                |                  |                 |
  |                |                  |---setItem('dark')---->|
  |                |<--theme='dark'---|                 |
  |<--UI updates---|                  |                 |
```

## Files to Modify

| File | Change | Description |
|------|--------|-------------|
| `styles/theme.css` | New | CSS variables for light/dark |
| `services/theme.ts` | New | Theme service with localStorage |
| `components/Settings.tsx` | Modify | Add toggle button |
| `app/layout.tsx` | Modify | Load theme on mount |
```

**Key rules**:
- Document rationale for EVERY major decision
- List alternatives considered (shows you thought it through)
- Include sequence diagrams for multi-step flows
- Specify exact files to modify

### sdd-tasks

**Purpose**: Break design into implementable tasks with dependencies

**Trigger**: When orchestrator launches you to create task breakdown

**What it does**:
1. Reads proposal, specs, and design
2. Breaks work into small, completable tasks
3. Groups tasks by phase (infrastructure, implementation, testing)
4. Uses hierarchical numbering (1.1, 1.2, etc.)
5. Identifies dependencies between tasks

**Artifacts created**:

**Engram mode**:
```typescript
mem_save(
  title: "sdd/{change-name}/tasks",
  topic_key: "sdd/{change-name}/tasks",
  type: "architecture",
  project: "{project}",
  content: "# Tasks: {change-name}\n\n..."
)
```

**OpenSpec mode**:
```
openspec/changes/{change-name}/tasks.md
```

**Tasks structure**:
```markdown
# Tasks: Dark Mode Toggle

## Phase 1: Foundation

- [ ] 1.1 Create `styles/theme.css` with CSS variables for light/dark
- [ ] 1.2 Create `services/theme.ts` with ThemeService class
- [ ] 1.3 Add theme types to `types/theme.ts`

## Phase 2: Integration

- [ ] 2.1 Add ThemeProvider to `app/layout.tsx`
- [ ] 2.2 Load theme from localStorage on mount
- [ ] 2.3 Apply theme class to body element

## Phase 3: UI Components

- [ ] 3.1 Create ToggleButton component in `components/ToggleButton.tsx`
- [ ] 3.2 Add toggle to Settings page at `app/settings/page.tsx`
- [ ] 3.3 Wire toggle to ThemeService.toggle()

## Phase 4: Testing

- [ ] 4.1 Unit tests for ThemeService
- [ ] 4.2 Component tests for ToggleButton
- [ ] 4.3 E2E test for theme persistence across sessions

## Phase 5: Polish

- [ ] 5.1 Add transition animation between themes
- [ ] 5.2 Add WCAG AA contrast check
- [ ] 5.3 Update documentation
```

**Key rules**:
- Group tasks by phase (don't mix infrastructure and UI)
- Keep tasks small (completable in one session)
- Use hierarchical numbering for clarity
- Phases must be completable in order (dependencies)

### sdd-apply

**Purpose**: Implement tasks by writing actual code

**Trigger**: When orchestrator launches you to implement one or more tasks

**What it does**:
1. Reads specs, design, and tasks
2. Detects TDD mode (if enabled)
3. Implements assigned tasks following specs
4. Marks tasks complete with `[x]` in tasks.md
5. Reports progress and deviations

**TDD Mode** (if enabled in config):
```
FOR EACH TASK:
1. RED   - Write failing test first
2. GREEN - Implement minimum code to pass
3. REFACTOR - Clean up without changing behavior
4. Mark task complete [x]
```

**Standard Mode**:
```
FOR EACH TASK:
1. Read task description
2. Read relevant spec scenarios
3. Read design decisions
4. Write the code
5. Mark task complete [x]
```

**Artifacts updated**:

**Engram mode**:
```typescript
// Update tasks with completion marks
mem_update(id: {tasks-observation-id}, content: "{updated tasks}")

// Save progress report
mem_save(
  title: "sdd/{change-name}/apply-progress",
  topic_key: "sdd/{change-name}/apply-progress",
  type: "architecture",
  project: "{project}",
  content: "{implementation progress}"
)
```

**OpenSpec mode**:
```
# Updates this file with [x] marks:
openspec/changes/{change-name}/tasks.md
```

**Key rules**:
- ALWAYS read specs before implementing
- ALWAYS follow design decisions (don't freelance)
- Match existing code patterns and conventions
- Mark tasks complete AS you go, not at the end
- If design is wrong, NOTE IT in return summary (don't silently deviate)
- In TDD mode, ALWAYS follow RED → GREEN → REFACTOR

**Trigger**: When orchestrator launches you to verify completed work

**Artifacts created**:

**Engram mode**:

)

**OpenSpec mode**:

**Verification checklist**:

## Spec Compliance

### Scenario: User enables dark mode
- [x] UI switches to dark theme
- [x] Preference saved to localStorage
- [x] Toggle shows "enabled" state

### Scenario: Dark mode persists across sessions
- [x] App loads in dark theme after reload
- [x] Toggle shows "enabled" state

- [x] Stored preference in localStorage as "theme" key

- Run tests if test infrastructure exists
- Note ANY deviations from design (even improvements)

### sdd-archive

**Purpose**: Archive completed change and merge deltas into main specs

**Trigger**: When orchestrator launches you to archive a verified change

**What it does**:
1. Reads proposal, specs, design, tasks, verify report
2. Merges delta specs into main specs (in `openspec/specs/`)
3. Moves change folder to archive with timestamp
4. Warns before destructive deltas (large removals)
5. Returns summary of what was archived

**Artifacts moved/updated**:

**OpenSpec mode**:
```bash
# Before:
openspec/changes/add-dark-mode/

# After:
openspec/changes/archive/2026-03-13-add-dark-mode/

# Specs merged:
openspec/specs/ui/theme-switching.md  # Updated with delta specs
```

**Engram mode**:
```typescript
// Archive metadata
mem_save(
  title: "sdd/{change-name}/archived",
  topic_key: "sdd/{change-name}/archived",
  type: "architecture",
  project: "{project}",
  content: "{archive summary with timestamp}"
)
```

**Key rules**:
- Warn before merging destructive deltas (large removals)
- Use ISO date format for archive folder: `YYYY-MM-DD-{change-name}`
- Merge delta specs into main specs (don't leave duplicates)
- Archive is an AUDIT TRAIL (never delete or modify archived changes)

## Shared Conventions

All SDD skills reference these shared convention files:

### persistence-contract.md

Defines mode resolution rules:

| Mode | Reads From | Writes To | Project Files |
|------|-----------|-----------|---------------|
| `engram` | Engram MCP | Engram | Never |
| `openspec` | Filesystem | Filesystem | Yes |
| `hybrid` | Engram + Filesystem | Both | Yes |
| `none` | Context only | Nowhere | Never |

See [SDD Component - Persistence Modes](/components/sdd#persistence-modes) for details.

### engram-convention.md

Defines Engram naming patterns:

```typescript
// Topic key structure:
"sdd/{change-name}/{artifact-type}"

// Examples:
"sdd/add-dark-mode/proposal"
"sdd/add-dark-mode/spec"
"sdd/add-dark-mode/design"
"sdd/add-dark-mode/tasks"
```

Using `topic_key` enables **upserts**—saving with the same key updates the existing observation instead of creating duplicates.

### openspec-convention.md

Defines filesystem structure:

```
openspec/
├── config.yaml
├── specs/{domain}/spec.md
└── changes/{change-name}/
    ├── proposal.md
    ├── specs/{domain}/spec.md
    ├── design.md
    ├── tasks.md
    └── verify-report.md
```

See [SDD Component - OpenSpec Directory Structure](/components/sdd#openspec-directory-structure) for full details.

## FAQ

<Accordion title="Can I skip phases in the SDD workflow?">
  Yes! The orchestrator can jump to any phase (e.g., `/sdd-apply` without specs). However, skipping phases reduces quality—you lose the "thinking" that specs and design provide.
</Accordion>

<Accordion title="What if I don't have Engram installed?">
  SDD falls back to `none` mode. Artifacts live only in conversation context. Install Engram or use `openspec` mode for persistence.
</Accordion>

<Accordion title="How do I enable TDD mode in sdd-apply?">
  Set `rules.apply.tdd: true` in `openspec/config.yaml`. The skill automatically follows RED → GREEN → REFACTOR.
</Accordion>

<Accordion title="What's the difference between sdd-explore and sdd-propose?">
  **sdd-explore** is research—understanding the problem space.
  
  **sdd-propose** is commitment—defining what you'll build.
  
  Explore is optional, propose is required.
</Accordion>

<Accordion title="Can I customize the SDD workflow?">
  Yes! Edit the skill files in your agent's skills directory, or add custom `rules` in `openspec/config.yaml` to enforce project-specific constraints per phase.
</Accordion>
